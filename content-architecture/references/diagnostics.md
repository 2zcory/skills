# Diagnostics

Use this file when the content problem is fuzzy and you need stronger diagnostic language.

## Common failure modes

- `equal-weight sprawl`: too many sections or blocks asking for the same attention
- `premature explanation`: the page explains structure before it earns trust or desire
- `proof weakness`: the page claims things before showing a visible consequence
- `route ambiguity`: there is no obvious next click or reading path
- `panel dependency`: every new idea becomes another box or card
- `return weakness`: the page has no live edge, freshness cue, or reason to revisit

## Questions that improve architecture fast

- What must the visitor believe after the first screen?
- What is the strongest proof available?
- What should be seen before navigation matters?
- Which section is doing duplicate work?
- What can move off this surface without harming understanding?
- If this became one column on mobile, would the reading order still make sense?

## Rewrite patterns

- Replace `three equal feature cards` with `one proof lane + two route rows`
- Replace `large supporting block` with `sidebar note + main content lane`
- Replace `long explanatory intro` with `one-sentence thesis + stronger first proof`
- Replace `bloated footer` with `closing direction + essential links only`

## Handoff prompts

- `Turn this blueprint into a mobile-first page layout with one dominant hero and sidebar support on desktop.`
- `Turn this architecture into a writing outline with one paragraph brief per section.`
- `Reduce this surface from seven sections to four without losing trust or route clarity.`
