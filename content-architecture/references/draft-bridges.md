# Draft Bridges

Use these patterns when a content architecture package is sound, but the next execution risk is weak translation from structure into concrete page language.

## Public home

Bridge pattern:

- `Opening line`: what kind of person, practice, or product this is in visitor language
- `Proof line`: one sentence that points to the strongest visible consequence
- `Route cue`: the clearest next click in plain language
- `Return cue`: one line that keeps the living edge visible

Example skeleton:

- Opening: `Follow a builder showing how unfinished ideas turn into shipped work.`
- Proof: `Start with one project that shows which decisions survived real constraints.`
- Route cue: `Go to Projects first, then Writing if you want the clearer position.`
- Return cue: `Come back through Garden when you want the work while it is still moving.`

## Project or case study

Bridge pattern:

- `Opening line`: what this project is and why it mattered
- `Pressure line`: the hard problem or constraint
- `Decision line`: the judgment that shaped the build
- `Next edge`: what still remains open

## Docs landing

Bridge pattern:

- `Opening line`: what this docs set helps you do
- `Fast-start line`: where to begin immediately
- `Reference line`: where deeper paths live
- `Maintenance cue`: what changed or what stays current

## Rules

- A draft bridge should reduce blank-page risk, not replace the architecture package.
- Prefer visitor language over internal route or framework vocabulary.
- If the bridge still sounds like a strategy memo, the architecture is not ready enough.
