# Page Archetypes

Use these defaults when a surface needs a fast architecture decision.

## Public home

Default sequence:

1. Opening
2. First proof
3. Route priority
4. Supporting depth
5. Closing direction

Notes:

- The opening establishes identity and the first trust signal.
- The first proof should not be a pile of equal tiles.
- Route priority should clarify the preferred next click.
- Supporting depth can be a lane, row set, or sidebar-supported section.
- The closing direction should keep momentum without turning into a generic footer dump.

## Project or case study

Default sequence:

1. Context
2. Pressure or hard problem
3. Key decisions
4. Outcome
5. Reflection or next edge

Notes:

- Put tradeoffs and judgment near the top.
- Avoid portfolio-summary language if one project can carry the proof better.

## Docs landing

Default sequence:

1. What this docs set is for
2. Fast start path
3. Reference paths
4. Update or maintenance signal

Notes:

- Do not make users read the taxonomy before giving them a first route.
- Use rows or grouped lists before card grids.

## App or dashboard overview

Default sequence:

1. Current state
2. Main task or decision lane
3. Supporting context
4. Deeper inspection paths

Notes:

- If the screen is operational, prefer utility language over marketing language.
- Sidebars should support the main task, not compete with it.
