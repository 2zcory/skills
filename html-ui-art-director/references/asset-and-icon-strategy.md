# Asset And Icon Strategy

Assets should support trust, orientation, or brand signal. If they do none of those, omit them.

## Icon policy

Decide one icon policy early:

- no icons
- sparse icons
- systematic icons
- expressive icons

Use one family and one stroke or fill logic per artifact.

Use icons when they:

- speed recognition
- distinguish repeated actions
- create compact orientation in dense tools

Avoid icons when they:

- repeat obvious labels
- add decorative noise
- create style conflict across sections

## Logo handling

When a real logo exists:

- give it a stable placement rule
- do not oversize it to compensate for weak composition

When no logo exists:

- reserve a clean placeholder mark slot if brand identity matters
- otherwise use a wordmark treatment or omit the mark entirely

## Imagery policy

Choose one:

- product imagery
- editorial imagery
- abstract shapes or patterns
- no imagery

Use imagery when it:

- proves the product exists
- builds trust
- provides atmosphere without fighting hierarchy
- explains the product faster than extra copy

Avoid imagery when it:

- fills blank space with no narrative role
- competes with the CTA
- reduces text clarity

## Placeholder handling

If the user did not provide assets:

- keep the slot explicit
- use neutral placeholder blocks, gradients, or pattern fields with clear intent
- state what asset would improve the final version

Do not pretend placeholders are final brand visuals.

## Background systems

Background treatment should match the visual language:

- calm neutral field
- soft gradient field
- grid or pattern field
- sectional contrast
- image-backed atmosphere

Backgrounds should create depth or mood, not compensate for weak layout.
