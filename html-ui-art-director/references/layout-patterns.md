# Layout Patterns

Choose the layout archetype from the screen job. The layout should answer what the user is here to do, not what is easiest to stack.

## Asymmetric Hero

Use when one message, one action, or one object should dominate the first view.

- Best for: landing pages, launches, feature intros, onboarding starts
- Avoid when: the screen is operationally dense

## Split Decision

Use when the user compares two states, two choices, or a message plus a live preview.

- Best for: editors, before/after, setup, AI generation, pricing comparisons
- Avoid when: one side has no real information value

## Rail And Canvas

Use when navigation, filters, or context should stay visible while the main work surface changes.

- Best for: dashboards, AI workspaces, admin tools, catalogs
- Avoid when: the rail becomes a graveyard of low-value controls

## Snapshot Plus Flow

Use when the user needs one top summary followed by a guided next action.

- Best for: personal dashboards, progress apps, account homes, task hubs
- Avoid when: the summary becomes a mini-dashboard with no clear next step

## Editorial Sequence

Use when persuasion, storytelling, or feature framing matters more than operational density.

- Best for: marketing sites, portfolios, product explainers
- Avoid when: every section has equal visual weight

## Control Cluster

Use when the task is tool-centric and the user needs grouped controls near the work object.

- Best for: settings, editors, builders, complex forms
- Avoid when: the controls bury the result or preview

## Layering rules

- Hero should dominate support.
- One viewport should tell the user what kind of place they are in.
- Quiet space is structural, not wasted.
- If support surfaces outnumber meaningful actions, merge or remove them.
- If every zone looks like a card, revisit the layout archetype first.

## Reset signals

Reset the layout instead of polishing it when:

- the first viewport has no obvious focal point
- support content visually beats the main action
- the screen reads as stacked modules rather than one composition
- mobile collapse turns the hierarchy flat

## Mobile rule

Do not merely stack desktop sections. Re-rank them.

On mobile:

- keep hero and next action near each other
- compress support content aggressively
- move low-value decoration behind the first decision
